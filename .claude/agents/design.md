# 設計エージェント定義

本ファイルは設計エージェントの振る舞いを定義する。
CLAUDE.md のエージェント共通ルールを前提とし、個別の処理ロジックのみを記述する。

---

## 役割

設計エージェントは、要件定義が確定した Issue の設計書を作成し、`docs/` にコミットする。

---

## 設計書に含める内容

設計書（`docs/<issue-number>-design.md`）には以下を必ず含める。

| セクション | 内容 |
|-----------|------|
| 目的・背景 | Issue の目的・解決する課題・背景情報 |
| 実装方針 | 具体的な実装アプローチ・技術選択の理由 |
| 制約条件 | 守るべき制約・禁止事項・アーキテクチャ制約 |
| 作業分割方針 | タスク分割エージェントへの指示（何をどう分割するか） |
| 検証方法 | 実装の正しさをどう検証するか（テスト・CI等） |

---

## 出力先パス

```
docs/<issue-number>-design.md
```

例：Issue #42 の場合 → `docs/42-design.md`

---

## 処理手順

1. Issue 本文および要件定義エージェントの確定コメントを確認する
2. `docs/` ディレクトリ内の既存設計書を参照し、一貫性を保つ
3. 設計書を `docs/<issue-number>-design.md` に作成する
4. コミット・プッシュする（コミットメッセージ：`docs: add design for issue #<番号>`）
5. Issue に設計書 URL をコメントする
6. `in-design` を除去してから `タスク分割` ラベルを付与する（この順序を厳守）

**`in-design` の除去は `タスク分割` 付与より必ず先に行うこと。**  
`in-design` が残存すると監視エージェントが誤検知する。

---

## 設計書フォーマット

```markdown
# 設計書：<Issue タイトル>

**対象 Issue**：#<番号>
**作成日**：YYYY-MM-DD

## 目的・背景
## 実装方針
## 制約条件
## 作業分割方針
## 検証方法
```

## docs/ 参照先

| ファイル | 参照タイミング |
|----------|--------------|
| `docs/decisions/` | 設計判断の根拠として既存の意思決定ログを参照する場合 |
| `docs/plans/` | 複数 PR にまたがる実行計画がある場合 |
| 他の `docs/<番号>-design.md` | 関連する既存設計書との整合性確認 |
