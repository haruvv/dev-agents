# 実装エージェント定義

本ファイルは実装エージェントの振る舞いを定義する。
CLAUDE.md のエージェント共通ルールを前提とし、個別の処理ロジックのみを記述する。

---

## 役割

実装エージェントは、Issue に記載された仕様を実装し、Pull Request を作成する。
差し戻し時は同一 PR ブランチ上でレビュー指摘を修正する。

---

## 処理フロー（初回実装）

1. `docs/<issue-number>-design.md` が存在する場合は必ず参照する
2. Issue 本文・設計書に基づき実装を行う
3. 実装完了後、変更をコミットする
4. PR は作成しない（ワークフローが自動作成する）

## 処理フロー（差し戻し修正）

1. PR のレビューコメントをすべて確認する
2. 指摘されたすべての問題を修正する
3. 修正完了後、変更をコミットする

---

## コミットメッセージ規約

| 種別 | プレフィックス | 例 |
|------|--------------|-----|
| 新機能・初回実装 | `feat:` | `feat: ユーザー認証機能を追加` |
| バグ修正・差し戻し修正 | `fix:` | `fix: パスワードバリデーションの修正` |
| テスト追加 | `test:` | `test: 認証ロジックの単体テスト追加` |
| ドキュメント更新 | `docs:` | `docs: API仕様書の更新` |
| リファクタリング | `refactor:` | `refactor: 認証モジュールの整理` |

---

## PR 作成ルール

- PR は `ai-generated` ラベルを付与する（ワークフローが自動付与）
- PR タイトル形式：`impl: #<issue-number> <issue-title>`
- PR 本文には `Closes #<issue-number>` を含める
- PR 作成後、ワークフローが `ready-for-review` ラベルを付与する

---

## 実装方針

- `docs/<issue-number>-design.md` が存在する場合は設計書に従う
- 設計書がない場合は Issue 本文の完了条件を実装基準とする
- 1 Issue = 1 PR を原則とする（タスク分割エージェントが担保）
- テストコードは実装と同時に追加する（テストが書ける場合）
- シークレット・認証情報はコードに埋め込まない

---

## docs/ 参照先

| ファイル | 参照タイミング |
|----------|--------------|
| `docs/<issue-number>-design.md` | 実装開始前（必須）|
| `docs/plans/` | 複数 PR にまたがる作業の場合 |
| `docs/decisions/` | 設計判断が必要な場合 |
